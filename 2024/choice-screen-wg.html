PROPOSED Choice Screen Working Group Charter

The mission of the Choice Screen Working Group is to set open, fair, and user-centric standards for choice screens so that people may pick the browser or search engine that best matches their preferences.

Motivation and Background

The power of defaults is a well-established force that shapes people's behavior: few users change default settings, their default browser, or their default search engine. In turn, these defaults shape the web at a structural level because they have a powerful influence over which browsers and which search engines people use.

Because defaults can be selected to favor the interests of the company that sets them rather than those of the user, a number of jurisdictions as well as some user-centric companies, are replacing defaults with choice screens. A choice screen is a dialog that prompts the user to make a deliberate decision as to which default should be applied. Ideally, this prompt happens rarely (perhaps only once) so as not to overwhelm people with constant decisions but it presents a good occasion to empower people to choose the option that best matches their preferences.

Unfortunately, experience shows that it is easy to produce a badly designed choice screen such that it will create other problems (e.g. favoring the richest company or the one that most readily sells personal data). As the number of jurisdictions in which choice screens are mandated grows (for browsers, search, or both), it becomes increasingly important to align on a recommended approach to choice screens.

Two facts are important. First, the web community now has real-world pragmatic experience with choice screens. This means that we have sufficient experience to inform the elaboration of a standard. And second, we can observe that the usage volume of a given browser or search engine determines how much power that browser or search engine has over the structure of the web. This means that there is an ethical necessity to make these volumes reflect user preferences.

Scope

The following is in scope for this group:

  • Definition of the user interface of choice screens, including accessibility considerations and the limited set of rubrics that may be used to help people make more informed choices.
  • Definition of the process through which a browswer or search engine can become a candidate in choice screens, including the dynamics of selection.
  • The operation of a registry of eligible browsers and search engines.

Normative Specifications

The Working Group will deliver the W3C normative specifications listed below. Note that because W3C is not (and should not be) in a position to preempt local regulations, these specifications are intended to provide a sensible global baseline for choice processes. Local jurisdictions may tighten or loosen the constraints chosen by the group. However, our hope is to provide enough of a credible baseline that it is almost always adopted as is.

Browser Choice Screen

This specification defines a choice screen for browsers. The choice screen displays a set of browsers from which the user can choose. Each browser has some standardized information listed, including possibly some labels drawn from the browser's registration and that describe properties that people struggle to assess for themselves (e.g. privacy properties).

The group may produce a reference implementation of this specification.

Browser Registry

This creates a registry of browsers eligible for inclusion in the Browser Choice Screen. Inclusion is based on a set of criteria defined by the group. Note: the group may decide to set a moderate fee for registration to cover the costs involved in verifying the criteria. This will not be set at a level that precludes entry by legitimate participants.

Search Choice Screen

This specification defines a choice screen for search engines. The choice screen displays a set of search engines from which the user can choose. Each search engine has some standardized information listed, including possibly some labels drawn from its registration and that describe properties that people struggle to assess for themselves (e.g. privacy properties).

The group may produce a reference implementation of this specification.

Search Registry

This creates a registry of search engines eligible for inclusion in the Search Choice Screen. Inclusion is based on a set of criteria defined by the group. Note: the group may decide to set a moderate fee for registration to cover the costs involved in verifying the criteria. This will not be set at a level that precludes entry by legitimate participants.

Success Criteria

In order to advance to Proposed Recommendation, each choice screen specification is expected to demonstrate its real-world viability. Because these are not standards focused on interoperable formats or protocols, viability is assessed not through a conformance test suite but by adoption. For the choice screen specifications to be ready for PR, they must be referenced normatively by two jurisdictions.

In order to advance to Candidate Registry Snapshot, each registry expected to demonstrate its real-world viability. Because this is not a standard focused on an interoperable format or protocol, viability is assessed not through a conformance test suite but by adoption. For the registries to be considered ready, their data must be referenced normatively by two jurisdictions.

There should be testing plans for each specification, starting from the earliest drafts. These will likely include user testing, internationalization testing, and accessibility testing.

Each specification should contain sections detailing all known security, privacy, and accessibility implications for implementers, Web authors, and end users.

Coordination

For all specifications, this Working Group will seek horizontal review for accessibility, internationalization, privacy, and security with the relevant Working and Interest Groups, and with the TAG. Invitation for review must be issued during each major standards-track document transition, including FPWD. The Working Group is encouraged to engage collaboratively with the horizontal review groups throughout development of each specification. The Working Group is advised to seek a review at least 3 months before first entering CR and is encouraged to proactively notify the horizontal review groups when major changes occur in a specification following a review.

Additional technical coordination with the following Groups will be made, per the W3C Process Document:

W3C Groups

The Working Group's deliverables are focused on strong alignment with the horizontal review groups. Those groups will be called upon systematically, but other groups need not be.

External Organizations

The Working Group should identify a set of regulators who are interested in choice screens and proactively collaborate with them where possible.

Participation

To be successful, this Working Group is expected to have 6 or more active participants for its duration, including representatives from the key implementors of this specification, and active Editors for each specification. The Chairs and specification Editors are expected to contribute half of a working day per week towards the Working Group. There is no minimum requirement for other Participants.

The group encourages questions, comments and issues on its public mailing lists and document repositories, as described in Communication.

The group also welcomes non-Members to contribute technical submissions for consideration upon their agreement to the terms of the W3C Patent Policy.

Participants in the group are required (by the W3C Process) to follow the W3C Code of Ethics and Professional Conduct.

Communication

Technical discussions for this Working Group are conducted in public: the meeting minutes from teleconference and face-to-face meetings will be archived for public review, and technical discussions and issue tracking will be conducted in a manner that can be both read and written to by the general public. Working Drafts and Editor's Drafts of specifications will be developed in public repositories and may permit direct public contribution requests. The meetings themselves are not open to public participation, however. The Working Group may decide to record its meetings and to make the video publicly available.

Information about the group (including details about deliverables, issues, actions, status, participants, and meetings) will be available from the Choice Screen Working Group home page.

Most Choice Screen Working Group teleconferences will focus on discussion of particular specifications, and will be conducted on an as-needed basis.

This group primarily conducts its technical work GitHub issues. The public is invited to review, discuss and contribute to this work.

The group may use a Member-confidential mailing list for administrative purposes and, at the discretion of the Chairs and members of the group, for member-only discussions in special cases when a participant requests such a discussion.

Decision Policy

This group will seek to make decisions through consensus and due process, per the W3C Process Document (section 5.2.1, Consensus). Typically, an editor or other participant makes an initial proposal, which is then refined in discussion with members of the group and other reviewers, and consensus emerges with little formal voting being required.

However, if a decision is necessary for timely progress and consensus is not achieved after careful consideration of the range of views presented, the Chairs may call for a group vote and record a decision along with any objections.

To afford asynchronous decisions and organizational deliberation, any resolution (including publication decisions) taken in a face-to-face meeting or teleconference will be considered provisional. A call for consensus (CfC) will be issued for all resolutions (for example, via email, GitHub issue or web-based survey), with a response period from one week to 10 working days, depending on the chair's evaluation of the group consensus on the issue. If no objections are raised by the end of the response period, the resolution will be considered to have consensus as a resolution of the Working Group.

All decisions made by the group should be considered resolved unless and until new information becomes available or unless reopened at the discretion of the Chairs.

This charter is written in accordance with the W3C Process Document (Section 5.2.3, Deciding by Vote) and includes no voting procedures beyond what the Process Document requires.
